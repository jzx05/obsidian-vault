# CaP-X: A Framework for Benchmarking and Improving Coding Agents for Robot Manipulation

## TL;DR
> CaP-X 是一个开源框架，用于系统性地评估和改进面向机器人操作的编码智能体（Coding Agents）。它包含三个核心组件：CaP-Gym（交互式仿真环境）、CaP-Bench（标准化评估基准）和 CaP-Agent0（无需训练的智能体框架）。此外还引入 CaP-RL，通过强化学习对编码智能体进行后训练，实现了 sim-to-real 迁移。

## 核心问题与动机

- **Code as Policies (CaP)** 范式：让 LLM/VLM 生成可执行代码来控制机器人，是 embodied AI 的一个重要方向
- **当前问题**：
  1. 现有 CaP 系统通常依赖高层级、硬编码的原语，难以适应精细操作需求
  2. 缺乏系统性的评估标准——不同抽象层级、感知模式、交互模式下的性能差异不明确
  3. 没有统一的训练框架来持续改进编码智能体
- **本文贡献**：提出 CaP-X，一个端到端的评估与改进框架，涵盖环境、基准、智能体设计和 RL 训练

## 框架整体架构

CaP-X 由四个核心模块组成：

| 模块 | 定位 | 功能 |
|------|------|------|
| **CaP-Gym** | 交互环境 | 多层级感知与控制原语的仿真平台，支持可验证的代码执行 |
| **CaP-Bench** | 评估基准 | 系统性评测，8 个难度层级（S1-S4 单轮 + M1-M4 多轮） |
| **CaP-Agent0** | 推理框架 | 无需训练的智能体，集成 VDM、技能库、并行推理 |
| **CaP-RL** | 训练范式 | 基于 GRPO 的强化学习后训练，利用 CaP-Gym 奖励信号 |

## CaP-Gym：交互式仿真环境

### 设计理念
- 基于 MuJoCo 构建，模拟标准桌面操作场景（Franka 机器人 + 桌面物体）
- 支持代码即策略的交互循环：智能体观察环境 → 生成 Python 代码 → 执行 → 获取反馈

### 感知原语（Perception Primitives）
- 通过可调用的感知接口获取结构化环境数据
- 观察结果为语义化 objects（物体位置、姿态、属性等）

### 控制原语（Control Primitives）
- **多层级抽象设计**，从高到低：
  - **High-Level（S1/S2）**：`pick(obj)`, `place(obj, pos)` 等语义化 API
  - **Low-Level（S3/S4）**：直接控制末端执行器位移、旋转，或通过轨迹点序列控制
- 低层级提供更强表达力，但生成难度更大
- 还包含工具函数：坐标变换、碰撞检测、路径规划辅助等

### 感知层级（Perceptual Grounding）
- **Privileged（S1/S3）**：直接提供精确物体位姿（oracle 信息）
- **Non-Privileged（S2/S4）**：仅提供 RGB 图像 + 视觉模型输出，模拟真实感知

## CaP-Bench：评估基准

### 评估维度

| 类别 | 特征 | 单轮 | 多轮 |
|------|------|------|------|
| **Perception** | Successive (Task-Based) | S1 | M1 |
| **Primitive Abstraction** | High-level / Low-level | S2, S3 | M2, M3 |
| **In-Context Learning** | Prompt/Large Examples | S4 | M4 |
| **Visual Grounding** | w/ or w/o VDM (YGSM) | - | - |

### 任务来源
- 从 7 个环境中筛选 87 个任务：CaP-Gym 自有任务 + 从 LIBERO-PRO、BEHAVIOR 改编的任务
- 涵盖：拾取放置、堆叠、旋转、双手操作、导航操作等

### 评估协议
- **Zero-Shot**：不提供示例，直接生成代码
- **单轮 (S1-S4)**：一次生成，执行评估
- **多轮 (M1-M4)**：允许迭代修正，基于执行反馈生成新代码
  - **Text-only Multi-turn (M1)**：仅文本反馈（stdout + error trace）
  - **Multimodal (M2)**：增加执行前后的 RGB 图像观察
  - **Low-Level with VDM (M3/M4)**：使用 Visual Differencing Model 将视觉差异转为结构化文本

### 评估层级详解（S1-S4 单轮 + M1-M4 多轮）

#### S1：特权状态、高层 API、单轮（Privileged Single-Turn）
- 模型可以调用人工设计好的**高层机器人操作函数**，且这些函数使用仿真环境的**真值状态**
- 真值信息包括：
  - 物体的真实位置和姿态
  - 真实的物体掩码
  - 无分割误差和位姿估计误差
  - 已封装好的抓取、移动等操作
- **代码示例**（抓起红色方块）：
```python
grasp_pos, grasp_quat = sample_grasp_pose("red cube")
open_gripper()
goto_pose(grasp_pos, grasp_quat, z_approach=0.1)
close_gripper()
goto_pose(grasp_pos + np.array([0, 0, 0.2]), grasp_quat)
```
- **测量能力**：
  - 是否理解自然语言任务
  - 是否能把任务分解成正确的操作顺序
  - 是否会正确调用高层机器人 API
  - **排除视觉感知误差后**，模型的任务规划上限
- **定位**：理想感知条件下的高层规划测试，人工脚手架最强、通常最容易的设置

#### S2：真实感知、高层 API、单轮（Non-Privileged Single-Turn）
- 仍提供人工设计的高层 API，但**不再使用仿真真值**
- 依赖实际视觉感知模块处理 RGB-D 图像，需面对：
  - 物体分割错误
  - 深度图噪声
  - 物体位姿估计误差
  - 抓取姿态估计失败
  - 遮挡和相似物体混淆
- 复杂的感知和控制流程仍被封装在高层函数中
- 模型主要负责决策：先找哪个物体 → 先抓什么 → 移动到哪里 → 何时开合夹爪 → 操作顺序
- **测量能力**：
  - 在真实、有噪声的视觉管线下调用高层技能的能力
  - 生成的操作序列是否能容忍感知误差
  - **S1 与 S2 的性能差距，可用于量化视觉感知贡献 vs 任务规划贡献**
- 论文指出，S2 比 S1 更接近以往 Code-as-Policies 方法的标准设置

#### S3：底层 API、有示例、单轮
- 开始移除人工编写的高层机器人技能
- 模型不能简单调用 `sample_grasp_pose()`、`goto_pose()`，而必须自己组合底层感知、几何和控制函数：
  - `get_observation()`
  - `segment_sam3_text_prompt(...)`
  - `point_prompt_molmo(...)`
  - `plan_grasp(...)`
  - `get_oriented_bounding_box_from_3d_points(...)`
  - `solve_ik(...)`
  - `move_to_joints(...)`
  - `open_gripper()` / `close_gripper()`
- **完整流程示例**（抓起红色方块）：
  1. `get_observation()` 获取 RGB、深度、相机内参和相机位姿
  2. 用 SAM3 或 Molmo 定位红色方块
  3. 从多个分割结果中选择正确掩码
  4. 把分割掩码和深度输入抓取规划器
  5. 选择抓取得分较高的抓取姿态
  6. 把抓取姿态从相机坐标系转换到世界坐标系
  7. 从旋转矩阵提取四元数
  8. 计算预抓取位置
  9. 调用逆运动学 `solve_ik()`
  10. 调用 `move_to_joints()` 移动机械臂
  11. 控制夹爪闭合
  12. 再次求解逆运动学并抬起物体
- **API 文档提供 `Example:` 使用示例**（如展示怎样提取 RGB、调用 SAM3、传参给抓取规划器）
- **测量能力**：
  - 能否组合底层感知和控制模块
  - 是否理解坐标系、旋转矩阵、四元数和逆运动学
  - 是否能正确处理数组形状、数据类型和函数返回值
  - **在提供示范代码情况下**，能否完成复杂的机器人程序合成
- 重点从"规划做什么"转向"如何做"

#### S4：底层 API、无示例、单轮
- 与 S3 完全相同的底层原子 API，但**删除了文档中的使用示例**
- 模型只能看到：函数签名、参数类型、返回值类型、普通 docstring、接口功能说明
- 模型看不到组合方法。例如它知道：
  - `segment_sam3_text_prompt(rgb, text_prompt)`
  - `plan_grasp(depth, intrinsics, segmentation)`
  - `solve_ik(position, quaternion_wxyz)`
- 但没有示例告诉它：
  - RGB 和 depth 应从观察字典的哪个键中提取
  - 应该选 SAM3 返回的哪个 mask
  - mask 是否需要转换数据类型
  - 抓取姿态位于相机坐标系还是世界坐标系
  - 应如何进行坐标变换
  - 四元数顺序是 wxyz 还是 xyzw
  - 如何把感知、抓取、IK 和运动控制串起来
- **测量能力**：
  - 能否仅根据 API 定义推断正确的程序结构
  - 是否真正理解机器人感知和控制接口，而不是模仿上下文示例
  - 能否从零合成完整的低层机器人控制程序
- **S4 是论文衡量模型单轮、零样本、底层机器人编程能力的核心设置**
- 论文发现：人类专家在 S4 条件下明显优于当前模型

#### M1：程序执行反馈、多轮、高层 API（Text-Only Multi-Turn）
- 可理解为：**S2 的有噪声感知和高层 API + 多轮执行与修改 + stdout/stderr 反馈**
- 每一轮模型生成代码后，环境执行代码，返回：
  - Python 标准输出 stdout
  - Python 错误信息 stderr
  - 异常堆栈
  - 模型自己插入的调试打印信息
- 模型不能直接看到环境图像，但可以主动在程序中加入：
```python
print(object_pose)
print(grasp_pos)
print(segmentation_scores)
print("Reached stage 2")
```
- 由此能在下一轮判断：程序是否异常、感知模块是否找到目标、位置估计是否合理、失败发生在哪个阶段、是否修改重试
- **测量能力**：
  - 根据执行日志进行调试
  - 主动添加状态探针
  - 根据异常和符号状态修复程序
  - 在没有直接视觉反馈的情况下进行失败恢复
- **重要**：多轮过程中**环境不会重置**。若第一轮已移动方块，第二轮面对的是移动后的场景

#### M2：原始 RGB 视觉反馈、多轮、高层 API
- 在 M1 的基础上，把**机器人当前观察到的 RGB 图像**直接放入编码模型的上下文
- 每轮模型同时看到：stdout + stderr + 当前环境的 RGB 图像
- 理论上，视觉语言模型可直接判断：
  - 物体是否被抓起
  - 方块是否堆叠成功
  - 夹爪是否抓空
  - 被操作物体当前位置
  - 场景与上一轮的变化
- 仅适用于**能接收图像输入的多模态模型**
- **反直觉结果**：直接给编码模型原始 RGB 图像，效果通常**比只给文本执行反馈的 M1 更差**
- 论文归因于**跨模态对齐问题**：模型可能分别擅长看图和写代码，但不一定擅长在同一上下文里联合完成视觉场景理解 + 机器人状态判断 + Python 错误调试 + 几何推理 + 后续代码生成
- **测量能力**：编码模型直接进行视觉闭环控制的能力，同时揭示原始图像未必是最有效的反馈形式

#### M3：视觉差分文字反馈、多轮、高层 API
- 引入论文的 **Visual Differencing Module (VDM)**，视觉差分模块
- 不直接把原始 RGB 交给编码模型，而是由专门的 VLM 先分析图像，再转换为**结构化自然语言**
- **第一轮**，VDM 负责描述：
  - 当前场景包含哪些相关物体
  - 物体的位置和空间关系
  - 与任务有关的视觉属性
  - 机器人和目标的当前状态
- **后续轮次**，VDM 重点描述：
  - 当前图像与上一轮图像有什么变化
  - 哪些物体被移动了
  - 抓取是否成功
  - 堆叠或插入是否成功
  - 任务是否已完成
  - 若未完成，可能出现了什么问题
- **编码模型获得**：stdout/stderr + VDM 场景文字描述 + 前后图像差异描述 + 任务完成状态判断
- 编码模型本身**不需要直接理解原始像素**
- **VDM 反馈示例**：
  > The red cube has moved upward but is no longer held by the gripper. It appears to have fallen to the right side of the table. The task is not complete.
- **测量能力**：
  - 根据结构化视觉文字进行闭环纠错
  - 结合程序日志和环境变化修改策略
  - 利用视觉差分判断动作是否达到预期效果
  - 在高层 API 条件下进行多轮任务恢复
- **重要发现**：M3 通常明显优于 M2 —— 把视觉信息**"语言化、结构化"**可能比直接塞入原始图像更有效

#### M4：VDM、多轮、底层 API
- 多轮配置中**最综合**的设置：
  - M3 的多轮执行和 VDM 反馈
  - + S3 的底层原子 API
  - + S3 的 API 使用示例
- **关键细节**：M4 对应 S3 的底层接口和使用示例，**不是 S4 的无示例设置**
- 模型需自己组合：SAM3/Molmo 目标定位、RGB-D 处理、3D 点云和物体几何计算、抓取生成、坐标系转换、逆运动学、关节空间运动、夹爪控制
- 每轮执行后获得：stdout/stderr + VDM 场景描述 + VDM 前后视觉差分 + 任务完成的视觉判断
- 第一轮程序即使在低层接口调用、分割、坐标转换或抓取规划中失败，模型也能在后续轮次诊断和修正
- **测量能力**：
  - 底层机器人程序合成
  - 多轮代码调试
  - 基于视觉结果进行物理失败恢复
  - 在较少人工高层封装的情况下，能否通过更多**测试时计算**弥补性能
  - 模型能否自己重新构建类似高层技能的程序结构
- **论文重要结论**：M4 虽使用底层 API，但通过多轮纠错和 VDM 视觉反馈，可**明显超过单轮底层 S3，甚至达到或超过单轮高层 S2**
- 这说明模型不必然依赖人类预先写好的高层机器人技能，也可通过多轮推理逐步获得稳健性

### CaP-Bench 实际包含的七项机器人任务

S1–S4 和 M1–M4 是**评测条件**。在这些条件下，论文主要评测以下七个操作任务。同一任务在不同 tier 下的**目标不变**，变化的是模型能获得什么信息、使用什么 API，以及能否根据执行结果继续修改代码。

| 任务 | 含义 | 核心能力 |
|------|------|---------|
| **Cube Lift** | 抓起指定方块 | 目标识别、抓取和抬升 |
| **Cube Stack** | 将多个物体尽可能叠高 | 顺序规划、重复抓放、稳定堆叠 |
| **Spill Wipe** | 用物体或工具擦除桌面污渍 | 非刚体视觉目标识别、轨迹覆盖 |
| **Peg Insert** | 将插销插入目标孔中 | 精确定位、姿态对准和插入 |
| **Cube Restack** | 按要求重新排列或堆叠方块 | 场景理解、关系推理、多步操作 |
| **Two-Arm Lift** | 用双臂共同抬起目标物体 | 双臂协调、同步抓取和运动 |
| **Two-Arm Handover** | 将物体从一只机械臂交给另一只 | 双臂时序、空间协调和交接 |

## 关键实验发现

### Takeaway 1：前沿模型与人类专家之间存在显著差距
- 单轮低层级任务中，人类参考实现 2-3 次迭代后成功率 ~50%
- 最佳模型（GPT-5.2）仅约 20%
- **说明编码智能体的规划和控制能力远未饱和**

### Takeaway 2：高层级抽象提升性能但限制表达力
- High-Level（S1/S2）成功率显著高于 Low-Level（S3/S4）
- 但高层级 API 无法表达精细动作（如旋转特定角度、沿特定路径移动）
- **权衡**：高层级降低了编程难度，但牺牲了策略的灵活性

### Takeaway 3：多轮交互 + 视觉差异化（VDM）显著提升性能
- 从单轮到多轮，所有模型成功率显著上升
- VDM 将原始图像对比转化为结构化文本描述，跨模型普遍 +10-15% 提升
- **结构化反馈比原始像素输入更有效**

### Takeaway 4：视觉 Grounding + 多模态输入进一步提升表现
- 非特权感知（需自行从图像中提取信息）下性能下降明显
- 加入视觉 grounding 后缩小差距，但无法完全弥补
- **结论**：纯视觉感知仍是瓶颈之一

## CaP-Agent0：无需训练的智能体框架

**CaP-Agent0 不是重新训练出来的机器人基础模型，而是套在现有 LLM/VLM 外面的 training-free 编码代理框架。** 它旨在解决 CaP-Bench 暴露的三个问题：

1. 单次生成的机器人程序容易出错，模型需要执行后再观察和修改
2. 直接把 RGB 图像交给编码模型，视觉和代码推理结合得不好
3. 底层机器人 API 很灵活，但模型每次都要重新实现坐标转换、点云处理、抓取选择等通用代码

### 整体闭环

```text
自然语言任务
      ↓
VDM 分析当前视觉场景
      ↓
多个编码模型并行生成候选方案（9 个候选）
      ↓
中央模型综合候选，生成最终 Python 程序
      ↓
在 CaP-Gym 沙箱中执行，控制仿真或真实机器人
      ↓
收集 stdout、stderr 和执行后的图像
      ↓
VDM 描述场景变化及任务是否完成
      ↓
多个模型判断结束还是继续，并生成修复程序
      ↓
重复，直到任务成功或达到轮数限制
```

CaP-Agent0 的 "agentic" 体现在：**观察环境 → 生成并执行代码 → 检查结果 → 根据失败修改 → 从多候选中选择或综合 → 复用以前发现的通用技能**。

### 模块一：Visual Differencing Module (VDM)

#### 第一轮：任务相关的场景描述
输入用户指令 + 当前 RGB → 输出结构化场景描述，例如：
> 场景中有红色、蓝色和黄色方块。黄色方块位于桌面中央，蓝色方块在它右侧。机械臂夹爪当前张开。任务要求将物体尽可能堆高。

帮助模型决定：哪个物体先作为底座、哪个先抓、是否需要避障、任务处于什么状态。

#### 后续轮次：变化 + 完成判断
同时查看上一张和当前图像，重点回答两个问题：
- 相比上一轮，场景发生了什么变化？
- 当前任务是否已完成？

> 蓝色方块已经从桌面右侧移动到黄色方块上方，但没有稳定地落在黄色方块上，目前向右倾斜。夹爪已经松开。任务尚未完成。

#### 为什么不直接给编码模型图片
CaP-Bench 的 M2 实验发现，直接把 RGB 塞入代码代理上下文性能反而下降。作者认为模型难以在同一上下文里同时完成 **图像理解 + 几何推理 + Python 编程 + 错误调试**。

**VDM 相当于视觉信息"翻译器"**：连续像素 → 与任务相关的结构化自然语言 → 编码模型更容易利用的反馈。

#### VDM 的局限
描述不保证完全正确，可能误判物体位置或错误宣布任务完成。所以代码代理仍需结合 stdout、感知结果和显式状态检查。

---

### 模块二：自动合成的持久技能库（Skill Library）

在 S3/S4 的底层 API 条件下，模型经常重复编写同一类通用函数（旋转矩阵转四元数、深度图转点云、mask 转世界坐标点、俯视抓取选择、轨迹插值等）。

#### 技能库的构建流程
```text
收集 12 个模型 × 7 个任务的成功 S3 rollout
→ 用正则提取其中的函数定义
→ 让 Gemini-3-Pro 找出频繁出现且任务无关的函数
→ 整理和验证
→ 得到 9 个任务无关技能
```

#### 九个通用技能

| 技能 | 作用 |
|------|------|
| `rotation_matrix_to_quaternion` | 3×3 旋转矩阵 → `wxyz` 四元数 |
| `decompose_transform` | 从 4×4 位姿矩阵提取位置和四元数 |
| `depth_to_point_cloud` | 深度图反投影为相机坐标系点云 |
| `mask_to_world_points` | 分割 mask 像素 → 世界坐标点集 |
| `pixel_to_world_point` | 单个像素及深度 → 世界坐标点 |
| `transform_points` | 对 3D 点集应用齐次变换 |
| `interpolate_segment` | 两个 3D 点之间生成轨迹航点 |
| `normalize_vector` | 三维向量归一化 |
| `select_top_down_grasp` | 从候选中选择俯视抓取 |

#### 技能库解决了什么问题
没有技能库时，模型每次抓取都得重写坐标变换、四元数转换、IK 求解等代码，容易出错的点包括：
- 忘记从相机坐标系转到世界坐标系
- 矩阵乘法顺序错误
- `xyzw` 与 `wxyz` 四元数顺序弄反
- 深度图维度不符、未过滤无效深度
- 选到侧向或倒置抓取
- mask、深度、点云形状不一致

技能库固定这些脆弱实现，让模型专注于更高层的问题：**抓哪个物体 / 放到哪里 / 操作顺序 / 失败后备用策略**。

#### 与人工高层 API 的关键区别
- **人工高层 API**（S1/S2）：机器人专家提前设计，包含任务特定先验，如 `sample_grasp_pose("red cube")` 直接解决目标定位和抓取规划
- **CaP-Agent0 技能库**：从模型自己的成功代码中自动提取，仅是任务无关的数据处理和几何工具，如 `mask_to_world_points()` 不决定要抓哪个物体

**在两个极端之间平衡**：保留底层 API 的表达力，同时避免重复实现脆弱代码，不把完整任务策略预先封装。

#### "持久" 的局限
理论上应该跨任务、跨 trial 持续积累新技能。但论文当前实现只做了**一次集中式技能合成**，没有展示长期在线自动更新。所以"持久" 更多是框架设计能力，而非已充分验证的长期学习结果。

### 模块三：并行推理与模型集成

每一轮并行生成 **9 个候选答案**，两种配置：

#### +1M：单模型集成
只用 Gemini-3-Pro，调用 9 次，温度分别为 0.1, 0.2, ..., 0.9。低温候选保守稳定，高温候选多样。

#### +3M：多模型集成
- Gemini-3-Pro / GPT-5.2 / Claude Opus 4.5
- 每个模型分别用温度 0.1, 0.5, 0.9
- 3 × 3 = 9 个候选

同时利用**同模型不同采样的多样性**和**不同模型的能力风格差异**。

#### 中央综合模型
不是多数投票、不是选最长的程序，而是把 9 个候选交给中央编码模型综合。提示词要求：
- 假设没有任何候选完全正确
- 对每个候选进行批判性检查
- 优先使用显式验证，而不是无依据假设
- 可以组合多个候选中的优点
- 根本冲突时选择更稳健方案

**示例**：候选 A 直接选最高分抓取；B 只选严格俯视；C 阈值放宽；D 回退最高分；E 增加预抓取位置。综合模型可组合为：
> 先选严格俯视 → 找不到时放宽方向阈值 → 仍失败时用最高分 → 执行预抓取、下降、闭合、抬升

#### 多轮阶段
每个候选模型独立判断 `FINISH` 或 `REGENERATE`，中央模型综合完成判断和修复建议。集成不仅提高成功率，也降低平均交互轮数——**模型集成经常在第一轮就预见潜在问题**，主动加入空分割、无效深度、抓取 fallback、IK 验证等防御性检查。

### Figure 8 消融实验（七项任务平均成功率）

| 配置 | 平均成功率 |
|------|:---:|
| **S3**：单轮 + 底层 API | 24% |
| **M4**：S3 + 多轮 VDM | 55% |
| **+SL**：M4 + Skill Library | 59% |
| **+1M**：技能库 + Gemini 九次并行 | 66% |
| **+3M**：三个模型并行集成 | **68%** |

关键观察：
1. **闭环交互带来最大单项提升**：S3→M4，+31%
2. **技能库增益温和**：M4→+SL，+4%
3. **并行推理显著加成**：+SL→+1M，+7%
4. **多模型 vs 单模型集成增幅小**：+1M→+3M 仅 +2%，但每轮 API 调用成本明显上升

### 分任务结果（+3M 完整配置）

| 任务 | 成功率 |
|------|:---:|
| Cube Lift | 97% |
| Cube Stack | 98% |
| Spill Wipe | 100% |
| Cube Restack | 89% |
| Two-Arm Lift | 74% |
| Two-Arm Handover | 20% |
| **Peg Insert** | **0%** |
| 平均 | 68.29% |

**边界揭示**：
- 视觉反馈 + 多轮恢复擅长解决抓放、堆叠、擦拭
- **Peg Insert = 0%**：视觉语言推理 + 代码纠错**无法弥补精密接触控制的不足**。需要毫米级对准、稳定力学接触和力/力矩反馈，CaP-Agent0 只有视觉 + 几何 + 位置控制，缺高频闭环接触策略
- **Two-Arm Handover = 20%**：双臂时序和交接仍很难

---

### 为什么称为 training-free

CaP-Agent0 没有针对七项任务做：模型微调 / 模仿学习 / RL / 机器人轨迹训练 / VLA 后训练。

它使用现成 Gemini、GPT、Claude，通过推理时机制提高性能：
```text
更多轮次 + 更多候选 + 视觉文字化 + 技能复用
```

属于 **test-time compute scaling** —— 不改权重，测试时投入更多推理和交互。

但仍依赖大量先验：预训练 LLM/VLM、SAM3、Molmo、Contact-GraspNet、PyRoki、离线提取的技能库、人类定义的底层 API 和成功条件。**更准确的理解是"不对编码基础模型做任务特定的参数训练"。**

---

### 学术观点

> 当模型面对底层机器人接口时，能力不足不一定全是权重知识不够，也可能是**单次推理预算不足、反馈形式不合适、通用程序结构无法复用**。

论文提出的替代路线：不是不断增加人工设计的高层机器人宏，而是让代理在底层接口上生成程序 → 执行观察 → 视觉和日志纠错 → 从成功代码提取可复用技能 → 多模型多采样提高覆盖率。

**中间路线**：
```text
极端一：人工写好所有高层技能（可靠但限表达力）
极端二：模型每次从最底层从零编程（灵活但错误率高）
CaP-Agent0：底层 API + 自动发现技能 + 多轮反馈 + 并行推理
```

---

### 需要谨慎看待的局限

- **计算成本高**：每轮 9 候选 + 综合模型 + VDM
- **延迟大**：不适合高频实时控制
- **依赖外部模块**：分割/抓取规划/IK 系统性失败时 LLM 难以修复
- **精密接触能力缺失**：Peg Insert = 0%
- **VDM 完成判断可能错误**
- **技能库并非真正实时成长**：当前是一次性离线合成
- **人类比较不严格等预算**：交互和推理预算不同
- **多模型收益/成本不成比例**：+1M→+3M 仅 +2%
- **部分高成功率依赖任务结构**：视觉上易判断结果的抓放/擦拭更适合此框架

---

### 一句话总结

> **CaP-Agent0 以 M4 为基础，用 VDM 把视觉执行结果转成文字，用自动技能库复用常见机器人几何代码，再用九路并行候选和中央模型综合生成更稳健的程序，从而在不微调基础模型的情况下，把底层 API 上的平均成功率从 S3 的 24% 提升到 68%。**

## CaP-RL：强化学习后训练

### 动机
- CaP-Agent0 依赖推理时计算（inference-time compute），成本高
- 能否通过 RL 训练让较小模型内化这些能力？

### 方法
- **算法**：GRPO（Group Relative Policy Optimization）
- **基础模型**：Qwen2.5-Coder-7B
- **奖励信号**：CaP-Gym 提供的可验证执行结果（编译通过、任务成功）
- **训练任务**：Code Lint, Code Save, Tool Type（专注代码质量和正确性）

### 训练细节
- Post-training 50 iterations per trial
- 每轮执行生成代码并获取 binary reward
- 重点优化：代码编译率、策略一致性、鲁棒性

### 关键结果

#### Sim-to-Real 迁移
- 在真实 Franka Emika 机器人上验证：
  - **Cube Lifting**：成功率 84%（仅在仿真中训练）
  - **Cube Stacking**：成功率 76%
- 证明 RL 训练的代码生成策略具备跨域泛化能力
- 模拟器中的奖励信号可有效指导真实机器人操作

#### 性能提升
- 编译成功率显著提高（减少语法/API 调用错误）
- 代码逻辑一致性增强
- 相比无 RL 基线，任务成功率提升明显

## Let the Agent Experience the Real World

### 真实环境部署
- CaP-Gym 环境支持与真实机器人对接
- 通过 ROS bridge 或直接的 Python 控制接口
- 智能体可以在真实环境中获取图像反馈，迭代修正代码
- **关键功能**：当代码执行失败时，机器人会自动回到安全位置

### 人机交互
- 支持人类在回路中给予额外反馈
- 通过自然语言补充约束或修正方向
- 适配 "报告实验进展 → 人类判断 → 继续/修改" 的工作流

## 与相关工作的对比

| 方法 | 类型 | 局限 |
|------|------|------|
| VLA (RT-2, π₀, Octo) | 端到端策略 | 缺乏可解释性，难以泛化到新任务 |
| Inner Monologue | CaP + 反馈 | 无系统评估，无多层级支持 |
| SWE-Bench | 代码 Agent 基准 | 面向软件工程，非机器人领域 |
| LIBERO | 操作基准 | 无代码生成评估，仅策略学习 |
| **CaP-X (本文)** | 完整框架 | 统一评估 + 训练 + 部署 |

## 核心洞察总结

1. **CaP 范式的价值**：相比端到端 VLA，代码策略具有可解释性、可组合性、可调试性——这些在长尾场景和新任务上至关重要
2. **抽象层级是关键设计选择**：需根据任务精度需求选择合适的控制抽象
3. **视觉理解仍是瓶颈**：将视觉信息结构化为文本是当前有效的桥接方案
4. **RL 后训练可行且有价值**：利用仿真环境的自动奖励，小模型也能学会有效的代码生成策略
5. **机器人操作是 Agent 能力的试金石**：需要规划、感知、推理、纠错的综合能力
