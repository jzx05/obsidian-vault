# CaP-X: A Framework for Benchmarking and Improving Coding Agents for Robot Manipulation

## TL;DR
> CaP-X 是一个开源框架，用于系统性地评估和改进面向机器人操作的编码智能体（Coding Agents）。它包含三个核心组件：CaP-Gym（交互式仿真环境）、CaP-Bench（标准化评估基准）和 CaP-Agent0（无需训练的智能体框架）。此外还引入 CaP-RL，通过强化学习对编码智能体进行后训练，实现了 sim-to-real 迁移。

## 核心问题与动机

- **Code as Policies (CaP)** 范式：让 LLM/VLM 生成可执行代码来控制机器人，是 embodied AI 的一个重要方向
- **当前问题**：
  1. 现有 CaP 系统通常依赖高层级、硬编码的原语，难以适应精细操作需求
  2. 缺乏系统性的评估标准——不同抽象层级、感知模式、交互模式下的性能差异不明确
  3. 没有统一的训练框架来持续改进编码智能体
- **本文贡献**：提出 CaP-X，一个端到端的评估与改进框架，涵盖环境、基准、智能体设计和 RL 训练

## 框架整体架构

CaP-X 由四个核心模块组成：

| 模块 | 定位 | 功能 |
|------|------|------|
| **CaP-Gym** | 交互环境 | 多层级感知与控制原语的仿真平台，支持可验证的代码执行 |
| **CaP-Bench** | 评估基准 | 系统性评测，8 个难度层级（S1-S4 单轮 + M1-M4 多轮） |
| **CaP-Agent0** | 推理框架 | 无需训练的智能体，集成 VDM、技能库、并行推理 |
| **CaP-RL** | 训练范式 | 基于 GRPO 的强化学习后训练，利用 CaP-Gym 奖励信号 |

## CaP-Gym：交互式仿真环境

### 设计理念
- 基于 MuJoCo 构建，模拟标准桌面操作场景（Franka 机器人 + 桌面物体）
- 支持代码即策略的交互循环：智能体观察环境 → 生成 Python 代码 → 执行 → 获取反馈

### 感知原语（Perception Primitives）
- 通过可调用的感知接口获取结构化环境数据
- 观察结果为语义化 objects（物体位置、姿态、属性等）

### 控制原语（Control Primitives）
- **多层级抽象设计**，从高到低：
  - **High-Level（S1/S2）**：`pick(obj)`, `place(obj, pos)` 等语义化 API
  - **Low-Level（S3/S4）**：直接控制末端执行器位移、旋转，或通过轨迹点序列控制
- 低层级提供更强表达力，但生成难度更大
- 还包含工具函数：坐标变换、碰撞检测、路径规划辅助等

### 感知层级（Perceptual Grounding）
- **Privileged（S1/S3）**：直接提供精确物体位姿（oracle 信息）
- **Non-Privileged（S2/S4）**：仅提供 RGB 图像 + 视觉模型输出，模拟真实感知

## CaP-Bench：评估基准

### 评估维度

| 类别 | 特征 | 单轮 | 多轮 |
|------|------|------|------|
| **Perception** | Successive (Task-Based) | S1 | M1 |
| **Primitive Abstraction** | High-level / Low-level | S2, S3 | M2, M3 |
| **In-Context Learning** | Prompt/Large Examples | S4 | M4 |
| **Visual Grounding** | w/ or w/o VDM (YGSM) | - | - |

### 任务来源
- 从 7 个环境中筛选 87 个任务：CaP-Gym 自有任务 + 从 LIBERO-PRO、BEHAVIOR 改编的任务
- 涵盖：拾取放置、堆叠、旋转、双手操作、导航操作等

### 评估协议
- **Zero-Shot**：不提供示例，直接生成代码
- **单轮 (S1-S4)**：一次生成，执行评估
- **多轮 (M1-M4)**：允许迭代修正，基于执行反馈生成新代码
  - **Text-only Multi-turn (M1)**：仅文本反馈（stdout + error trace）
  - **Multimodal (M2)**：增加执行前后的 RGB 图像观察
  - **Low-Level with VDM (M3/M4)**：使用 Visual Differencing Model 将视觉差异转为结构化文本

### 评估模型
- 覆盖 12 个前沿模型：GPT-5.2, Claude Opus 4.5, Gemini-3-Pro, GPT-o4-mini, o3-mini, Claude Sonnet 4.5, Deepseek-R1, GPT-OSS-120B, Qwen3-235B 等

## 关键实验发现

### Takeaway 1：前沿模型与人类专家之间存在显著差距
- 单轮低层级任务中，人类参考实现 2-3 次迭代后成功率 ~50%
- 最佳模型（GPT-5.2）仅约 20%
- **说明编码智能体的规划和控制能力远未饱和**

### Takeaway 2：高层级抽象提升性能但限制表达力
- High-Level（S1/S2）成功率显著高于 Low-Level（S3/S4）
- 但高层级 API 无法表达精细动作（如旋转特定角度、沿特定路径移动）
- **权衡**：高层级降低了编程难度，但牺牲了策略的灵活性

### Takeaway 3：多轮交互 + 视觉差异化（VDM）显著提升性能
- 从单轮到多轮，所有模型成功率显著上升
- VDM 将原始图像对比转化为结构化文本描述，跨模型普遍 +10-15% 提升
- **结构化反馈比原始像素输入更有效**

### Takeaway 4：视觉 Grounding + 多模态输入进一步提升表现
- 非特权感知（需自行从图像中提取信息）下性能下降明显
- 加入视觉 grounding 后缩小差距，但无法完全弥补
- **结论**：纯视觉感知仍是瓶颈之一

## CaP-Agent0：无需训练的智能体框架

### 整体架构
```
Task Description + API Docs + Skill Library
         ↓
   Parallel Reasoning (多模型并行生成)
         ↓
   Candidate Code → Synthesis → Final Code
         ↓
   Python Sandbox 执行
         ↓
   Runtime Outcome → VDM Feedback → 下一轮迭代
```

### 核心组件

#### 1. Visual Differencing Model (VDM)
- 输入：执行前后的两帧 RGB 图像
- 输出：结构化文本描述，说明场景发生了什么变化
- 例如："红色方块从 (0.3, 0.2) 移动到了 (0.5, 0.4)，蓝色方块未移动"
- **作用**：将视觉信息编码为 LLM 可直接理解的格式，替代原始像素

#### 2. 自动合成与持久化技能库（Skill Library）
- 成功执行的代码自动提取为可复用函数
- 泛化处理：将具体物体名称参数化，保留通用逻辑
- 持久化存储，后续任务可直接调用，避免重复推理
- **类似人类程序员积累的代码库**

#### 3. 并行推理与代码合成
- 同时向多个前沿模型发送请求（GPT-5.2, Claude Opus 4.5, Gemini-3-Pro）
- 各模型独立生成候选代码
- 由一个集成智能体（synthesis agent）选择/融合最佳方案
- **降低单模型的随机性和盲区**

#### 4. 迭代优化循环
- 执行失败后，收集错误信息 + VDM 反馈
- 智能体分析失败原因，生成改进版代码
- 多轮迭代直至成功或达到预算上限

### CaP-Agent0 性能
- 在 CaP-Bench 上超过 100 个任务的评测中，显著优于所有单模型 baseline
- 关键提升来自：VDM 结构化反馈 + 并行推理 + 技能库复用

## CaP-RL：强化学习后训练

### 动机
- CaP-Agent0 依赖推理时计算（inference-time compute），成本高
- 能否通过 RL 训练让较小模型内化这些能力？

### 方法
- **算法**：GRPO（Group Relative Policy Optimization）
- **基础模型**：Qwen2.5-Coder-7B
- **奖励信号**：CaP-Gym 提供的可验证执行结果（编译通过、任务成功）
- **训练任务**：Code Lint, Code Save, Tool Type（专注代码质量和正确性）

### 训练细节
- Post-training 50 iterations per trial
- 每轮执行生成代码并获取 binary reward
- 重点优化：代码编译率、策略一致性、鲁棒性

### 关键结果

#### Sim-to-Real 迁移
- 在真实 Franka Emika 机器人上验证：
  - **Cube Lifting**：成功率 84%（仅在仿真中训练）
  - **Cube Stacking**：成功率 76%
- 证明 RL 训练的代码生成策略具备跨域泛化能力
- 模拟器中的奖励信号可有效指导真实机器人操作

#### 性能提升
- 编译成功率显著提高（减少语法/API 调用错误）
- 代码逻辑一致性增强
- 相比无 RL 基线，任务成功率提升明显

## Let the Agent Experience the Real World

### 真实环境部署
- CaP-Gym 环境支持与真实机器人对接
- 通过 ROS bridge 或直接的 Python 控制接口
- 智能体可以在真实环境中获取图像反馈，迭代修正代码
- **关键功能**：当代码执行失败时，机器人会自动回到安全位置

### 人机交互
- 支持人类在回路中给予额外反馈
- 通过自然语言补充约束或修正方向
- 适配 "报告实验进展 → 人类判断 → 继续/修改" 的工作流

## 与相关工作的对比

| 方法 | 类型 | 局限 |
|------|------|------|
| VLA (RT-2, π₀, Octo) | 端到端策略 | 缺乏可解释性，难以泛化到新任务 |
| Inner Monologue | CaP + 反馈 | 无系统评估，无多层级支持 |
| SWE-Bench | 代码 Agent 基准 | 面向软件工程，非机器人领域 |
| LIBERO | 操作基准 | 无代码生成评估，仅策略学习 |
| **CaP-X (本文)** | 完整框架 | 统一评估 + 训练 + 部署 |

## 核心洞察总结

1. **CaP 范式的价值**：相比端到端 VLA，代码策略具有可解释性、可组合性、可调试性——这些在长尾场景和新任务上至关重要
2. **抽象层级是关键设计选择**：需根据任务精度需求选择合适的控制抽象
3. **视觉理解仍是瓶颈**：将视觉信息结构化为文本是当前有效的桥接方案
4. **RL 后训练可行且有价值**：利用仿真环境的自动奖励，小模型也能学会有效的代码生成策略
5. **机器人操作是 Agent 能力的试金石**：需要规划、感知、推理、纠错的综合能力
